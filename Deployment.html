Deployment in SFDC

Salesforce Deployment is nothing but copying/transferring Salesforce metadata from one Salesforce organization to another. There are various methods to achieve Salesforce metadata deployment.

Change Sets

It is the easiest way to send configuration changes from one Salesforce organization to another. But the change set also has a limitation that we can only use it among connected Salesforce orgs only (like Developer Sandbox to Test Sandbox & Test Sandbox to Production Org etc.). We cannot use it in between any individual Salesforce orgs.

To use change sets we need to have two Salesforce org’s i.e. one Production org and its Sandbox org.

  • Production Org: It is Salesforce org with live data which is being actively used.
  • Sandbox:A sandbox is a copy of your production organization. Sandboxes contain the same metadata as production organization.

Outbound Change Sets:

An outbound change set is a change set created in Source organization and that you want to deploy to target organization.

Sending an outbound change set to another organization doesn’t guarantee that the changes will be implemented in that organization. The change set must be deployed (accepted) by the target organization before the changes take effect.

Inbound Change Sets:

An inbound change set is a change set that is sent from source organization to the target organization. A change sent must be deployed for the changes to take effect. You can deploy the contents of an inbound change set as a whole, but not on a component-by-component basis.

Accept, modify, or reject change sets uploaded from other organizations.

Change sets require a deployment connection between the two Salesforce organizations.

The first step is to create this connection.

Steps of deployment by using ChangeSet:

In PRODUCTION Org 

  • First we have to check there is a connection between Production org (Source Org) and Sandbox Org (Destination Org).To check this 
  • Deployment settings (in setup) 
  • Click on edit near the sandbox from which deployment has to be done.
  • Then click on the checkbox named as “allow inbound changeset”.

In SANDBOX Org 

  • Go to Deployment settings (in setup)
  • Check “Allow deployments of components when corresponding Apex jobs are pending or in progress.
  • Caution: Enabling this option may cause Apex jobs to fail.” checkbox is tick or not. If not then make it checked.

OutBound Changeset (For a Specific Object - Sandbox)

  • Create New and add components like
  • Custom Object
  • Custom Tab
  • Custom Field (Except Standard Field like Name)
  • Lightning Page (Record Page)
  • Apex Class
  • Aura Component Bundle (Lightning Component/Custom Page) 
  • Process Builder/Flow/Work-Flow/Validation Rule
  • List View (ALL/Recent View)
  • Page Layout
  • Record Type
  • Permission Sets

Upload

In PRODUCTION Org

1. InBound Changeset (In Setup)

  • Allow to Sandbox option
  • Open the Inbound Changeset (Which is uploaded from Sandbox)
  • Validate (3 Types)

                1. Components other than Apex Classes - Run as Default

                2. Components of Apex class - Run All Test class (Code Coverage Should be more than 75% for each Class)

                3.If uploading only Specific Apex class - Run as Specific Test Class (Give test className on description)

2. Deployment Status (Check the Status)

3. Deploy

   - Deploy with the Test Classes as "Run All" or "Run As Specific" if required.

NOTE

  • In Production org we can't allow to change any code in Apex class, for that we need to change in Sandbox and again deploy the specific Apex Class with Test Class also and code coverage should be more than 75% 
  • In Production we can change codes of VisualForce Page & Aura Components.
  • In Production  we can create Aura & LWC or can create in VS Code and deploy but it is not a best practice.
